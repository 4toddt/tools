## Regenerating the per-SKU CSV

```bash
cd docs/projects/bcc-cover-sku-reconciliation
.xlsx-venv/bin/python scripts/build-by-sku-csv.py
```

If `.xlsx-venv` is missing, recreate it once:

```bash
python3 -m venv .xlsx-venv && .xlsx-venv/bin/pip install openpyxl
```

The script writes `data/cover-skus-by-color.csv` and prints a summary (row count, drops, issue breakdown, and a B153808 spot-check).

### What the CSV does and doesn't try to be perfect about

- **Color line parsing** uses `^(\d{2})\s*[=\-]\s*(.+?)\s*$`. Trailing-spaces-for-wrap and the few `code - name` rows are tolerated. Anything that fails to parse lands in `issues=unparsed-color-line` rather than being dropped silently.
- **Legends decoded:** wearability (`N/M/P`), specialty fabric (`I/C/A/PF/N/W*B`), leather dot (`A/P/N`), leather product line (`SLR/SLS/CLM/CLP/SL`), specialty leather (`A/C/I`). Fabric product line codes (`CH/DC/MU`) are preserved verbatim because the workbook has no authoritative mapping for them.
- **Drops:** `dropStatus=dropped` when the per-color rich-text run has strikethrough or red `FF0000`. The cell-level signal is binary; the workbook's distinction between "Phase Out / New Phase Out / April 2026 Drop" lives only in the legend block and is not recoverable per row.
- **Married Style strikethrough on leather** (a separate drop signal for specific married skus) is not surfaced in `dropStatus`; the raw `marriedStyles` text is preserved verbatim.
- **`issues` column flags:** `unparsed-color-line`, `hyphen-separator`, `duplicate-series` (same series appears on more than one workbook row), `formatting-ambiguous` (rich-text runs split mid-line). Treat issues as triage prompts, not errors.
